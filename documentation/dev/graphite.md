# Graphite backend (developer guide)

Graphite is Skia's next-generation GPU backend. It targets modern GPU APIs (Vulkan, Metal, Dawn / WebGPU) and uses an explicit Recorder/Recording/Submit model — different from Ganesh's auto-flush draw stream.

This guide covers the SkiaSharp public surface, the headless Lavapipe development story, and what's in v1 vs deferred.

---

## TL;DR — drawing a frame

```csharp
using SkiaSharp;

// 1. Bring up Vulkan (or Metal, or Dawn) yourself, then wrap the handles.
using var bc = new SKGraphiteVkBackendContext {
    VkInstance         = vkInstance,
    VkPhysicalDevice   = vkPhysicalDevice,
    VkDevice           = vkDevice,
    VkQueue            = vkQueue,
    GraphicsQueueIndex = graphicsQueueFamilyIndex,
    MaxApiVersion      = 0x00403000u,                      // VK_API_VERSION_1_3
    GetProcedureAddress = (name, instance, device) =>
        device   != IntPtr.Zero ? vkGetDeviceProcAddr  (device,   name) :
        instance != IntPtr.Zero ? vkGetInstanceProcAddr (instance, name)
                                : vkGetInstanceProcAddr (IntPtr.Zero, name),
};

// 2. Create a Graphite context. After this returns, `bc` may be disposed —
//    SKGraphiteContext takes ownership of the GetProc GCHandle (Variant A).
using var ctx = SKGraphiteContext.CreateVulkan (bc);
if (ctx == null) throw new InvalidOperationException ("Graphite/Vulkan create failed");

// 3. Each thread / frame gets a fresh recorder.
using var recorder = ctx.CreateRecorder ();

// 4. Make a Graphite-backed surface and draw with the SAME SKCanvas API
//    you use for Ganesh / raster.
var info = new SKImageInfo (1024, 1024, SKColorType.Rgba8888, SKAlphaType.Premul);
using var surface = SKSurface.Create (recorder, info);
var canvas = surface.Canvas;
canvas.Clear (SKColors.White);
using var paint = new SKPaint { Color = SKColors.Red, IsAntialias = true };
canvas.DrawCircle (512, 512, 256, paint);

// 5. Snap → insert → submit. The recording is consumed; dispose it.
using (var recording = recorder.Snap ())
    ctx.InsertRecording (recording);
ctx.Submit (new SKGraphiteSubmitInfo { Sync = SKGraphiteSyncToCpu.Yes });

// 6. Read back pixels (uses Context.asyncRescaleAndReadPixels under the hood —
//    SkSurface.ReadPixels is GPU_TEST_UTILS-only on Graphite-backed surfaces).
var pixels = new byte[info.BytesSize];
unsafe {
    fixed (byte* p = pixels)
        ctx.ReadPixels (surface, info, (IntPtr)p, info.RowBytes, 0, 0);
}
```

## Public surface (v1)

| Type | Status | Notes |
|---|---|---|
| `SKGraphiteContext` | ✅ Vulkan only | `CreateVulkan` / `IsBackendAvailable` / `CreateRecorder` / `InsertRecording` / `Submit` / `ReadPixels` / `CheckAsyncWorkCompletion` / disposal + budget controls |
| `SKGraphiteRecorder` | ✅ | `Snap` returns nullable. Threading: single-thread-affine; multiple recorders per context may be used in parallel on different threads. |
| `SKGraphiteRecording` | ✅ | Disposable. Insert via `SKGraphiteContext.InsertRecording`. Reusing after insertion is undefined. |
| `SKGraphiteVkBackendContext` | ✅ | `IDisposable` class (see [Variant A lifetime](#variant-a-lifetime) below). Properties for instance/device/queue/getProc. |
| `SKGraphiteContextOptions` | ✅ ⚙️ | Auto-generated value struct (multisample count, budget, etc.). |
| `SKGraphiteSubmitInfo` | ✅ ⚙️ | Auto-generated. `Sync`, `MarkBoundary`, `FrameID`. |
| `SKGraphiteInsertRecordingInfo` | ✅ ⚙️ | Auto-generated. Optional target surface + translation/clip. |
| `SKGraphiteBackend` enum | ✅ ⚙️ | `Dawn = 0`, `Metal = 1`, `Vulkan = 2`. |
| `SKGraphiteInsertStatus` enum | ✅ ⚙️ | Six values. Returned by `InsertRecording` — non-`Success` is not an exception. |
| `SKSurface.Create(SKGraphiteRecorder, SKImageInfo, …)` | ✅ | New overloads. Original Ganesh / raster overloads untouched. |

⚙️ = auto-generated by `pwsh ./utils/generate.ps1` from the C headers.

## Variant A lifetime

`SKGraphiteContext.CreateVulkan(bc)` takes ownership of:

- The `GCHandle` pinning the `GetProcedureAddress` delegate.
- The native `sk_graphite_vk_backend_context_t*` wrapper.

After a successful call returns, the caller may safely `bc.Dispose()` immediately. The Skia `Context`'s Vulkan dispatch lambda captured the C function pointer + userData by value — it no longer needs the wrapper. The Vulkan handles themselves (instance/device/queue) are never freed by SkiaSharp.

This **differs from `GRVkBackendContext`** — Ganesh's lifetime rule still requires the backend-context to outlive the `GRContext`. We did Graphite right because we're greenfield.

---

## Disposal ordering

Same convention as everywhere else in SkiaSharp: **dispose dependents before parents**. Canonical safe order is `recording → surface → recorder → context`.

`Dispose` is idempotent on every Graphite type. Disposing in the WRONG order (context first) is undefined behavior (likely native-side crash) — same as `GRContext` with live `SKSurface`s.

## Headless development story (Linux, no GPU)

SkiaSharp's CI uses **Mesa Lavapipe** — the CPU-only software Vulkan ICD shipped by `mesa-vulkan-drivers`. Lavapipe is fully deterministic, so visual tests get bit-exact pixel comparison.

```bash
sudo apt-get install -y mesa-vulkan-drivers libvulkan-dev vulkan-tools
vulkaninfo --summary  # expect a device named "llvmpipe" or "lavapipe"

CC=clang CXX=clang++ \
dotnet cake --target=externals-linux --arch=x64

dotnet test tests/SkiaSharp.Tests.Console/SkiaSharp.Tests.Console.csproj \
  --filter "FullyQualifiedName~Graphite|FullyQualifiedName~Visual"
```

See [`graphite-headless.md`](graphite-headless.md) for the full bring-up + troubleshooting notes.

## Test layers

There are **three concentric layers** of Graphite tests, each gated on its own pass criterion. If a layer fails, the inner layer's success isolates the bug to that layer.

| Layer | Location | What it proves |
|---|---|---|
| C smoke | `tests/native/Graphite/c_smoke/` (CMake + clang) | Vulkan/Lavapipe + Skia Graphite + the `sk_graphite_*` C shim work end-to-end. Bisects build / native-shim bugs from binding bugs. |
| C# smoke | `tests/Tests/SkiaSharp/Graphite/GraphiteSmokeTests.cs` | The full P/Invoke + managed-wrapper stack works. |
| Visual tests | `tests/Tests/SkiaSharp/Visual/` | Backend-agnostic drawing produces stable, byte-comparable pixels. New tests need only a draw lambda. See [Adding visual tests](#adding-visual-tests). |

Plus two backend-correctness regression suites:

- **Coexistence** (`GraphiteCoexistenceTests`) — proves `GRContext` + `SKGraphiteContext` work in the same process without interfering.
- **Disposal** (`GraphiteDisposalTests`) — idempotent dispose, canonical-order safety.

## Adding visual tests

```csharp
public class MyVisualTests : VisualTestBase
{
    [SkippableTheory, MemberData (nameof (AllSetups))]
    public void MyTest (VisualSetup setup) =>
        VerifyDraw (setup, nameof (MyTest), new SKImageInfo (256, 256), canvas => {
            // pure drawing; no backend-specific code
        });
}
```

First run produces a "no golden" failure with the rendered output saved to `_failures/{setup}/MyTest.actual.png`. If it looks correct:

```bash
SKIASHARP_UPDATE_GOLDENS=1 dotnet test --filter MyTest
```

Goldens land at `tests/Tests/SkiaSharp/Visual/Goldens/{setup}/MyTest.png`, ready to commit. Subsequent runs compare bit-for-bit (per-channel tolerance 2 by default — override via `MaxChannelDelta`).

`MemberData` source filters: `AllSetups`, `GpuSetups`, `RasterSetups`, `GraphiteSetups`, `GaneshSetups`. Adding new GPU setups is "extend `Setups/` + register in `VisualTestBase` member data", no per-test changes.

## Two non-obvious facts that shaped this code

These bit during integration; they're worth remembering before reaching for the obvious solution:

1. **`SkSurface::readPixels` is `#if defined(GPU_TEST_UTILS)` only on Graphite-backed surfaces.** `src/gpu/graphite/Device.cpp:onReadPixels` returns `false` in production builds. The supported readback path is `Context::asyncRescaleAndReadPixels` + `Context::checkAsyncWorkCompletion`. The C API mirrors those primitives directly; the synchronous `SKGraphiteContext.ReadPixels` is a managed-side composition that spins on completion.
2. **`ContextFactory::MakeVulkan` does NOT auto-create a `VulkanMemoryAllocator`** the way Ganesh's `GrVkGpu` does. Our shim's `sk_graphite_context_make_vulkan` calls `skgpu::VulkanMemoryAllocators::Make` internally when one isn't supplied (always, in v1) — transparent to callers.

Both are documented in `specs/002-graphite-backend-support/contracts/c-api.md` § "Discoveries during implementation".

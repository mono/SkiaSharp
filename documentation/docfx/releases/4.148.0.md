# Version 4.148.0

> [NuGet](https://www.nuget.org/packages/SkiaSharp/4.148.0)

> **Supersedes [4.147.0](4.147.0.md)** · Rolls up preview-only work that was never released as stable — those changes are included cumulatively below.

> **API changes** · [SkiaSharp API diff](4.148.0/index.md) · [HarfBuzzSharp 14.2.0](harfbuzzsharp/14.2.0.md)

**SkiaSharp `4.148.0` is the first stable release of the `v4` line.** It rolls up the three `4.147.0` previews (which were never published as stable) plus the new work in Release Candidate 1 — an upgrade to Skia milestone `m148`, HarfBuzz `14.2.0`, real variable-font and color-palette support, animated WebP encoding, a rebuilt device/WASM test harness on `dotnet test`, and the removal of the pre-v4 obsolete API surface. Because `v4` is a major bump you should read the [Breaking Changes](#breaking-changes) section below carefully — most compile-time breaks are the promotion of long-obsolete APIs to error, but there are also behavioral changes on `SKFont`/`SKPaint`/`SKTypeface` and an `SKStream.Move(long)` overflow behavior change that no signature diff can show. HarfBuzz `14.2.0` ships alongside — see [HarfBuzzSharp 14.2.0](harfbuzzsharp/14.2.0.md) for the binding details.

## Breaking Changes

`v4` is a **major** upgrade — several breaks land in this release. Most of them are compile-time obsolete APIs promoted to error (things you'd have seen deprecation warnings for during the `2.x`/`3.x` era). Two categories require attention that a signature diff cannot show: behavioral changes and interop-struct layout changes. See the [SkiaSharp API diff](4.148.0/index.md) for the exact compile-time signature list.

**Compile-time signature breaks (visible in the API diff):**

- **`GRVkImageInfo.YcbcrConversionInfo` retype** from `GrVkYcbcrConversionInfo` → `GRVkYcbcrConversionInfo` and **loss of value equality** on that type ([#3741](https://github.com/mono/SkiaSharp/pull/3741)) — hosts of managed Vulkan render targets must construct the new struct and stop relying on `Equals`/`==` semantics.
- **`SKNativeObject` setter removal** — the previously-obsolete public setter is gone in `v4`. Handle lifetime is fully managed by the runtime object.
- **`SKPaint` text/font members** and remaining pre-`v4` obsoletes have been promoted from warning to **error** ([#4068](https://github.com/mono/SkiaSharp/pull/4068), [#4114](https://github.com/mono/SkiaSharp/pull/4114), [#4205](https://github.com/mono/SkiaSharp/pull/4205)) and obsolete enum values have been **trimmed from the ref assembly**. Move text/font state to `SKFont`.

**Behavioral breaks (same signature, different runtime behavior):**

- **`new SKFont()` now has an empty typeface** ([#3735](https://github.com/mono/SkiaSharp/pull/3735)) — was `null` and lazily resolved at draw time in `3.x`; in `4.x` it is `SKTypeface.Empty` (0 glyphs) so `MeasureText` returns `0` and nothing is drawn. Construct with an explicit typeface, e.g. `new SKFont(SKTypeface.Default)`, and use `font.Typeface.IsEmpty` in place of `font.Typeface == null`.
- **`new SKPaint().Typeface` returns Default instead of null** ([#3736](https://github.com/mono/SkiaSharp/pull/3736)) — SkiaSharp seeds a usable default so text still renders, and `Reset()` restores it. The only observable break is null-checks: use `paint.Typeface.IsEmpty` (or drop the check).
- **`SKTypeface.FromFamilyName("missing")` returns Default, not null, on Android** ([#3737](https://github.com/mono/SkiaSharp/pull/3737)) — all platforms are consistent in `v4`. For a strict "was the font actually found?" check use `SKFontManager.Default.MatchFamily(name, style)` which still returns `null` on no match.
- **`SKTypeface.Default` now resolves through the font manager** ([#3738](https://github.com/mono/SkiaSharp/pull/3738)) — guaranteed non-null on every platform, and Android API 36+ (where the manager can itself return null) falls back to `SKTypeface.Empty` instead of crashing. No migration needed.
- **`SKStream.Move(long)` now throws and is obsolete** ([#3740](https://github.com/mono/SkiaSharp/pull/3740)) — the native offset is 32-bit and the old overload silently truncated large offsets. The overload is `[Obsolete]` (**CS0618**) and the cast is `checked((int)offset)` so out-of-range values now throw `OverflowException`. Switch to `Move(int)` and clamp/validate large offsets yourself.

**Interop / native-struct breaks (not in any API diff):**

- **Vulkan `GRVkBackendContextNative` — five fields removed** ([#3741](https://github.com/mono/SkiaSharp/pull/3741)): `fMinAPIVersion`, `fInstanceVersion`, `fExtensions`, `fFeatures`, `fOwnsInstanceAndDevice`. Delete those assignments; use `fVkExtensions` in place of `fExtensions`, and `fDeviceFeatures` / `fDeviceFeatures2` in place of `fFeatures`. Skia no longer manages instance/device ownership through the backend context, so `fOwnsInstanceAndDevice` has no replacement.
- **Vulkan `GrVkYcbcrConversionInfo` layout change** ([#3741](https://github.com/mono/SkiaSharp/pull/3741)): a new `Components` field (and a new `GRVkYcbcrComponents` struct) was added, changing the struct layout. Code that manually constructs the struct or relies on its layout must account for the new field.

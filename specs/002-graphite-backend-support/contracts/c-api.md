# Contract: Graphite C API

**Date**: 2026-04-30 · **Last amended**: 2026-05-01 · **Branch**: `002-graphite-backend-support`

This contract defines the C-level shim that the C# binding will P/Invoke into. It is the durable interface boundary between SkiaSharp's managed and native sides.

> **Status legend**:
> ✅ Shipped in MVP (US1, smoke-gate green) · 🔶 Deferred to US2 (Metal/Dawn backends) · 🔷 Deferred to US3 (BackendTexture wrap, Image-Wrap, AsImage*) · ⚙️ Hand-written shipped + auto-generated by `pwsh ./utils/generate.ps1` for P/Invoke side.

---

## Discoveries during implementation (2026-05-01)

Two facts about upstream Skia surfaced during integration that shape the contract and would otherwise be re-debated by future contributors:

1. **`SkSurface::readPixels` does not work on Graphite-backed surfaces in production builds.** `externals/skia/src/gpu/graphite/Device.cpp:onReadPixels` is `#if defined(GPU_TEST_UTILS)` only; the production fallthrough returns `false`. The supported readback path is `Context::asyncRescaleAndReadPixels` + `Context::checkAsyncWorkCompletion`. The C API mirrors those primitives directly (see § "Async readback"). Sync semantics are composed in the **C# layer** (busy-wait on completion); the C/C++ side stays a 1:1 pass-through of upstream Skia.
2. **`ContextFactory::MakeVulkan` does not auto-create a `VulkanMemoryAllocator`.** Unlike Ganesh's `GrVkGpu::Make` (which builds a default VMA allocator under `#if defined(SK_USE_VMA)`), Graphite's Vulkan factory rejects a `VulkanBackendContext` with no `fMemoryAllocator` set. Our shim solves this **transparently to callers**: `sk_graphite_context_make_vulkan` calls `skgpu::VulkanMemoryAllocators::Make` (private Skia API, in-tree) when the caller didn't supply one. Consequence: the C API's `sk_graphite_vk_backend_context_init_t` does NOT expose an allocator field.

`SK_GRAPHITE`, `SK_DAWN`, `SK_METAL`, `SK_VULKAN` come from the SkiaSharp build defines. `SK_GRAPHITE` and `SK_DAWN` are added in this feature; `SK_VULKAN` and `SK_METAL` already exist. The `:core` GN target (where the C shim sources compile) gets these defines explicitly via the SkiaSharp-injected block in `externals/skia/BUILD.gn` — Skia's own `:graphite` target carries them via `all_dependent_configs`, but `:core` doesn't inherit through that chain (analogous to the existing handling for `SK_GANESH`/`SK_VULKAN`). The `_vulkan/_metal/_dawn` translation units MUST be excluded from the build when their backend flag is off (FR-018).

## Common conventions

- Function naming: `sk_graphite_<type>_<verb>[_<modifier>]`.
- Opaque handles: `typedef struct sk_graphite_<type>_t sk_graphite_<type>_t;` defined as forward declarations.
- Enum members follow the existing SkiaSharp suffix convention (`VALUE_TYPENAME`, e.g. `VULKAN_SK_GRAPHITE_BACKEND`), so the SkiaSharp generator can strip the typename suffix to produce idiomatic C# enum names (`SKGraphiteBackend.Vulkan`).
- Failure: nullable factory functions return `NULL`; status-style functions return `false` or a status enum.
- Ownership transfer is encoded in the function name suffix:
  - `_new` — caller owns the returned pointer; pair with the matching `_delete`.
  - `_acquire` — caller takes a ref on a ref-counted Skia object.
  - No suffix on returned-borrow functions; documented inline.
- All structs marshalled across the boundary are POD (no virtuals, no inheritance), 8-byte aligned, packed into the same struct definition C# will pin/copy.
- Strings: C-strings (`const char*`) for labels; ownership stays with the caller.
- **No SkiaSharp-specific inventions on the C surface.** Every entry point maps 1:1 to a public method or free function in upstream Skia (the only adapter logic is the lambda in `make_vulkan` capturing `getProc`+`userData`, and the `AsyncReadCallbackBridge` for the async-read callback ABI). The "convenience-sync ReadPixels" lives in the managed layer, not here.

## ABI / stability rules

- Existing `sk_*` headers MUST NOT be edited. New types only, new functions only.
- Struct field additions in v1.x patch releases are FORBIDDEN (additions go to a new `_v2` struct).
- Each function prototype is decorated with `SK_API` for export visibility, matching the rest of the C shim.
- All callbacks (`sk_graphite_release_proc_t`) MUST be `cdecl`-equivalent (default on supported platforms; explicit `SK_C_API_CC` on Windows).

## Test obligations (informs Phase 2 / `/speckit-tasks`)

- The C smoke test exercises every function listed under "Core" and at least one backend's full set; backends without a host runner are exercised at the linker level only (compiled, not loaded).
- Each function whose return is nullable on failure MUST have a test that triggers the failure path (e.g., `make_render_target` with zero size).
- Each `_delete` function is fuzzed against `NULL` (must be a no-op).
